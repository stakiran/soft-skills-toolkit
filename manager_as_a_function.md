# 機能としてのマネージャー(Manager As A Function)
- マネージャーを機能とみなし、メンバーが必要に応じてこれを使う。略称は MaaF。
- 役割:
    - マネージャー: プロジェクトやチームをマネジメントする者。1人
    - メンバー: マネージャーの配下に属する者。n人
- メリット:
    - Internal DevRel の推進。メンバーのニーズに応えることを優先できる
    - 継続的な改善。機能のつくりかたや見せ方という形で改善していける

# Tools
MaaF を理解して試してみるためのワークリスト:

```markdown
- 1: MaaF と聞いて、思い浮かべたことを書いて
    - (書いて)
    - ...
- 2: 下記の「例」を読んで
- 3: MaaF についての理解度を書いて
    - (書いて)
    - ...
- 4: MaaF を使う価値があるかどうかを考えて
    - (y|n を選択して)
    - その理由: (理由を書いて)
```

MaaF として何を提供するかを考えるためのチェックリスト:

```markdown
- 1: Direct Appointment を提供しますか？
- 2: AMA を使いますか？
- 3: RMA を使いますか？
- 4: オフィスアワーを使いますか？
- 5: マネージャー・チケットを使いますか？
```

MaaF をどう提供するかを考えるためのトリガーリスト:

```markdown
- モードの設計と表示はどうやる？
- メンバー全員にいきなり展開する？それともまずは特定メンバーを選んで試す？
- メンバーからの依頼は即時に答える？時間差を許容してもらう？
- あなたが提供できそうな「主な提供方法」は何パターンある？
```

AMA をどのように提供するかを考えるためのトリガーリスト:

```markdown
- 同期的？非同期的？
- 1on1？少人数？もっと多数？
- 質問は一つずつもらって答える？複数まとめてもらったのにまとめて答える？
- 上記について、あなたの好みは？
- 上記について、メンバーの好みは？
```

マネージャー・チケットを設計するための質問リスト:

```markdown
- 1: チケット1枚分の「作業時間」「作業内容」「アウトプット」は？
- 2: チケットはどのスパンで何枚発行する？
- 3: チケットの発行や回答はどのツール上で行う？
- 4: チケット対応中に割り込みが来たときの対応はどうする？
- 5: チケット対応に関する状態（受付中・対応中・停止中等）はどのように可視化または通知する？
```

# 例: マネージャーがメンバー向けに提示する「モード」
- MaaF モード
    - MaaF として動く
- Solo モード
    - 個人ワークを行う
    - MaaF もしないし、ミーティングもしない
- Default モード
    - MaaF でも Solo でもない通常の状態
    - マネージャーなのでミーティングが多いか、必要に応じて行う

# 例: マネージャーが提供可能なインターフェース
- Direct Appointment
    - マネージャーの予定表の空いているところに、誰でも自由に予定を追加していいとする
- AMA(Ask Me Anything)
- [RMA(Request Me Anything)](rma_request_me_anything.md)
- オフィスアワー
- マネージャー・チケット
    - チケットとは「マネージャーに 1 時間分の検討と回答をお願いできる」もの
    - マネージャーはチケットをメンバーに発行し、使ってもらう
    - チケットによる作業は非同期的に行い、回答はメッセージやドキュメントの形で行う。プロトタイプなど実装をしても良いが必須ではない
